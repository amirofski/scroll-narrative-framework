# Philosophy

Software changes.

Experiences remain.

SNF is built on the belief that an interactive experience is a product of ideas rather than implementation details.

The purpose of the framework is not to provide another animation library.

Its purpose is to create a universal language for describing experiences.

An experience should outlive the technologies used to render it.

React will evolve.

Browsers will evolve.

Animation libraries will evolve.

Rendering technologies will evolve.

The experience itself should remain stable.

---

# Experience as the Primary Artifact

Traditional frontend development treats source code as the primary artifact.

SNF rejects this assumption.

In SNF, the primary artifact is the experience.

Source code is merely one possible implementation.

Developers author experiences.

The compiler transforms them.

The runtime executes them.

---

# Intent over Implementation

Authors should describe what should happen.

The engine determines how it happens.

Example:

Instead of writing:

* create a GSAP timeline
* register ScrollTrigger
* calculate offsets
* synchronize video playback

An author should simply express:

"The hero section fades while the video progresses and the next scene becomes visible."

The engine determines the optimal execution strategy.

Intent should never depend on implementation.

---

# Declarative by Default

Every experience should be expressed declaratively.

Declarative descriptions are:

* easier to understand
* easier to validate
* easier to optimize
* easier to serialize
* easier to version
* easier for AI systems to generate

Imperative code should exist only inside engine internals.

---

# Compiler-Centric Design

The compiler is the intelligence of the system.

Runtime complexity should be minimized.

The compiler is responsible for:

* parsing
* validation
* optimization
* graph generation
* dependency analysis
* asset analysis
* static error detection
* execution planning

The runtime should execute plans rather than interpret author intent.

---

# Data Is More Valuable Than Code

Behavior should emerge from data.

Experiences should be represented as immutable data structures.

Data can be:

* analyzed
* optimized
* cached
* serialized
* diffed
* tested
* visualized
* generated by AI

Executable logic should consume data instead of replacing it.

---

# Signals over Events

Events represent moments.

Signals represent continuous state.

Modern interactive experiences are primarily driven by continuous state.

Scroll position.

Video progress.

Mouse movement.

Audio analysis.

Sensor data.

Time.

All of these are signals.

Signals continuously influence experience state.

Events become implementation details.

---

# Progress Is Only One Signal

Scroll is not special.

Time is not special.

Mouse is not special.

Everything is a signal.

Signals may originate from:

* scroll
* time
* pointer
* keyboard
* touch
* media
* sensors
* multiplayer
* external APIs
* AI systems

The scheduler evaluates signals without knowing their origin.

---

# Graphs over Timelines

Timelines describe sequences.

Experiences are not always linear.

Real experiences contain:

* branches
* dependencies
* synchronization
* conditions
* parallel execution
* reusable structures

A graph models these relationships naturally.

Execution should therefore be graph-based.

---

# Composition over Inheritance

Capabilities should be composed.

Inheritance introduces coupling.

Composition encourages reuse.

Every feature should be attachable, removable and replaceable.

Plugins extend composition.

They do not modify the core.

---

# Runtime Independence

The engine should never depend on a specific rendering technology.

Core packages must remain independent from:

* Browser APIs
* DOM
* React
* Vue
* GSAP
* Three.js
* Canvas
* WebGPU

These belong to adapters.

Not the engine.

---

# Immutable Architecture

Immutable structures improve predictability.

Compiler outputs should be immutable.

Execution graphs should be immutable.

Experience definitions should be immutable.

Mutable state belongs only to runtime execution.

---

# Predictability over Magic

Magic reduces clarity.

The framework should behave consistently.

Every transformation performed by the compiler should be understandable.

Every optimization should be deterministic.

Developers should be able to inspect the entire compilation process.

---

# Explicit over Implicit

Hidden behavior creates confusion.

Engine behavior should always be explicit.

Defaults are acceptable.

Hidden assumptions are not.

---

# AI-Native Development

SNF is designed for a future where AI contributes to software development.

Architecture documents should be readable by machines.

Specifications should become executable knowledge.

Every architectural decision should exist as structured documentation.

AI should never infer architecture from source code alone.

The repository itself should explain the system.

---

# Plugins as First-Class Citizens

The engine should remain intentionally small.

Everything else belongs in plugins.

Examples include:

* Lottie
* Video
* Audio
* Three.js
* Canvas
* Rive
* Analytics
* Accessibility
* Motion Reduction

The plugin system should be stable enough that the ecosystem can evolve independently from the core.

---

# Experiences Should Be Portable

The same experience definition should execute across multiple runtimes.

Different rendering technologies should produce equivalent behavior.

Portability is a core objective rather than an optional feature.

---

# Testability

Every experience should be testable without a browser.

Compiler outputs should be validated independently.

Execution graphs should be inspectable.

Runtime behavior should be reproducible.

Deterministic systems are easier to test than imperative animation code.

---

# Simplicity Is an Architectural Goal

Simple does not mean minimal.

Simple means understandable.

Complexity should be isolated.

Developers should experience a small API surface while the engine handles internal complexity.

---

# Long-Term Thinking

Architectural decisions should prioritize the next ten years rather than the next release.

Temporary implementation shortcuts should never become permanent architectural constraints.

The engine should be capable of supporting technologies that do not yet exist.

---

# The Role of the Compiler

The compiler is responsible for transforming ideas into executable plans.

The runtime is responsible for executing those plans.

The renderer is responsible for displaying the result.

Each layer has one responsibility.

Responsibilities should never overlap.

---

# The Role of the Author

Authors create narratives.

They should not think about animation engines.

They should not think about rendering pipelines.

They should not think about browser scheduling.

They should think about storytelling.

The engine exists to translate stories into execution.

---

# Closing Statement

SNF is not an animation framework.

It is not a rendering engine.

It is not a visual editor.

SNF is an Experience Composition Engine.

Its responsibility is to transform human intent into deterministic, optimized and portable interactive experiences.

Everything in the architecture exists to serve that single purpose.
